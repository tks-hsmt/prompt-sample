# Helm Chart スキル

Helm chart の作成・修正・レビューを、`references/` 配下の組織ルールに準拠して行う。

すべての作業は **ヒアリング → 概要計画提示 → 承認 → 詳細計画作成 → 実行 → 検証** の順で進める。
承認を得るまでファイルの作成・編集は行わない。

## モード判定

ユーザーの依頼内容から作業モードを判定する。判断がつかない場合はユーザーに確認する。

| ユーザーの意図 | モード |
|---|---|
| 新しい chart を作りたい | 作成 |
| 既存の chart を修正・変更したい | 修正 |
| chart をレビュー・チェックしたい | レビュー |

---

## 作成モード

### ヒアリング

chart を構成するために必要な情報をユーザーから聞き出す。一度にすべてを聞かず、回答内容に応じて追加の質問を行う。

最初に確認する項目:
- チャート名
- デプロイ対象アプリケーションの概要（何をするアプリか）
- ワークロード種別（Deployment / DaemonSet / StatefulSet / Job / CronJob）
- コンテナイメージのリポジトリとタグ/ダイジェスト

ワークロード種別に応じて追加で確認する項目:
- 公開ポート、Service タイプ、Ingress、Egress の要否
- ヘルスチェック方式（HTTP パス、TCP、コマンド）
- レプリカ数、オートスケーリングの要否
- 永続ボリュームの要否（StorageClass、サイズ）
- 環境変数、設定ファイルの有無
- リソース要求・制限値（CPU / メモリ）
- 外部リソースへのアクセス要否（AWS, GCP 等 → ServiceAccount / RBAC の判断材料）

ヒアリングでは以下を心がける:
- ユーザーが回答に迷っている場合はベストプラクティスに基づく推奨値を提案する
- 「わからない」「おまかせ」と言われた場合はリファレンスのルールに従ったデフォルト値を採用し、計画で明示する
- 不足情報があればその都度追加で質問する。推測で埋めない

### 概要計画の提示

ヒアリング内容をもとに、作成する chart の全体像を概要計画として提示する。
概要計画は `results/helm-chart-plan-{yyyyMMddHHmm}/overview.md` に保存する。

概要計画に含める内容:
- チャート名、ワークロード種別
- 生成するファイル一覧（ディレクトリ構成）
- `values.yaml` の主要キーと値
- 環境別オーバーライド（`values-dev.yaml` / `values-stg.yaml` / `values-prod.yaml`）の差分方針
- セキュリティ設定（securityContext、RBAC）
- 特記事項（ユーザー指定値がルールと異なる場合の注記）

概要計画提示後、ユーザーに承認を求める。修正要望があればヒアリングに戻る。

### 詳細計画の作成

承認後、ファイルの作成に入る前に、作成するファイルごとの詳細計画を作る。詳細計画の作成手順は「詳細計画の作成手順」セクションを参照。

### 実行

詳細計画に従って chart ファイルを作成する。詳細計画にリファレンスのルールが反映済みのため、実行時にリファレンスを読み直す必要はない。詳細計画の順序（01 → 10）でファイルを作成する。

### 検証

「検証手順」セクションに従って検証を実行する。

---

## 修正モード

### ヒアリング

以下を確認する:
- 対象チャートのパス
- 変更したい内容

対象チャートを読み込み、現状を把握したうえで変更の影響範囲を分析する。影響範囲が広い場合や変更意図が曖昧な場合は追加で質問する。

### 概要計画の提示

概要計画は `results/helm-chart-plan-{yyyyMMddHHmm}/overview.md` に保存する。

概要計画に含める内容:
- 変更対象ファイルと変更内容の一覧
- 変更による影響範囲
- ルール違反が検出された場合はその指摘と修正案

概要計画提示後、ユーザーに承認を求める。

### 詳細計画の作成

承認後、変更対象ファイルごとの詳細計画を作る。変更対象以外のファイルの詳細計画は不要。詳細計画の作成手順は「詳細計画の作成手順」セクションを参照。

### 実行

詳細計画に従ってファイルを編集する。詳細計画にリファレンスのルールが反映済みのため、実行時にリファレンスを読み直す必要はない。

### 検証

「検証手順」セクションに従って検証を実行する。

---

## レビューモード

### ヒアリング

必要に応じて以下を確認する:
- 対象チャートのパス

### レビュー実行

判定は対象チャートの現在の内容とリファレンスのルールのみに基づいて行う。`results/` 配下の過去のレビュー結果や計画ファイルは読み込まない。過去の判定に影響されず、毎回独立した評価を行うことが正確なレビューには不可欠である。

レビュー結果の保存先は `results/helm-chart-review-{yyyyMMddHHmm}.md` とする。`{yyyyMMddHHmm}` 部分は Bash で `date +%Y%m%d%H%M` を実行して取得した現在日時で置き換える。ファイル作成前に必ず日時を取得すること。

#### 実行の流れ

リファレンスファイルを一括で読み込まない。以下の手順でリファレンスを **1 つずつ** 処理する。

1. 対象チャートの全ファイルを読み込む
2. レビュー結果ファイルを作成し、ヘッダー（対象チャート、日時、観点）を書き込む
3. 以下をリファレンスファイルごとに繰り返す:
   a. リファレンスファイルを **1 つだけ** 読み込む
   b. そのリファレンスのルールに基づいてチャートを検査する
   c. 判定テーブルと違反の修正ブロックをレビュー結果ファイルに **追記** する
   d. 次のリファレンスに進む
4. すべてのリファレンスの処理が完了したら、サマリーテーブルを末尾に追記する

処理するリファレンスの順序:
1. `general-conventions.md`
2. `chart-files.md`
3. `values.md`
4. `templates.md`
5. `labels.md`
6. `pods-basic.md`
7. `pods-security.md`
8. `pods-reliability.md`
9. `rbac-basic.md`
10. `rbac-least-privilege.md`
11. `values-environment.md`
12. `documentation.md`

#### レビュー結果のフォーマット

レビューの目的は **すべての違反を漏れなく可視化し、各違反に修正案を提示すること** である。違反の省略・要約・後回しは一切行わない。

レビュー結果には、リファレンスファイルごとにそのルールへの準拠状況を以下の3段階で判定して記載する:

- **準拠**: ルールに適合している
- **違反**: ルールに違反している（修正が必要）
- **対象外**: 対象チャートにそのルールが適用されない（例: RBAC を使用していないチャートでの RBAC ルール）

#### 出力の流れ

レビュー結果は **リファレンスファイルを 1 つずつ順に処理** して出力する。1 つのリファレンスの出力が完了してから次のリファレンスに進む。全リファレンスの処理が完了したら末尾にサマリーテーブルを置く。

リファレンスごとの出力手順:
1. そのリファレンスに含まれるルールを 1 行 1 ルールでテーブルに書く（判定・該当箇所・備考を含む）
2. テーブルの直後に、**違反と判定したルールすべて** について修正ブロックを書く。違反がなければ修正ブロックは不要
3. 次のリファレンスに進む

以下のテンプレートに従って出力する:

```markdown
# Helm Chart レビュー結果

- 対象チャート: `<chart-path>`
- レビュー日時: YYYY-MM-DD HH:MM
- レビュー観点: <全般 / セキュリティ / 構造 等>

## general-conventions

| # | ルール | 判定 | 該当箇所 | 備考 |
|---|---|---|---|---|
| 1 | チャート名はハイフン区切りの小文字 | 準拠 | Chart.yaml:1 | |
| 2 | バージョンは SemVer 2 | 違反 | Chart.yaml:3 | 下記参照 |
| 3 | YAML インデントはスペース 2 | 準拠 | — | |

**違反 2: バージョンが SemVer 2 に準拠していない**

- **重要度**: 必須修正

**現在の記述:**
\`\`\`yaml
version: 1.0
\`\`\`

**修正イメージ:**
\`\`\`yaml
version: 1.0.0
\`\`\`

**理由:** SemVer 2 では MAJOR.MINOR.PATCH の 3 桁が必須。

## サマリー

| リファレンス | 準拠 | 違反 | 対象外 |
|---|---|---|---|
| general-conventions | 2 | 1 | 0 |
| chart-files | X | X | X |
| ... | ... | ... | ... |
| **合計** | **X** | **X** | **X** |
```

#### 省略禁止ルール

- 違反と判定したルールは **1 件も省略せず** 修正ブロックを書く。件数が多くても「残りは同様」「省略」としない
- 修正ブロックには必ず **現在の記述** と **修正イメージ** のコードブロックを含める。「上記と同様の修正」としない
- サマリーテーブルの違反数と、各リファレンスセクションの修正ブロック数は一致しなければならない

レビュー結果提示後、ユーザーが修正を希望する場合は修正モードに切り替える。

## 詳細計画の作成手順

作成モード・修正モード共通で、概要計画の承認後・実行前に詳細計画を作成する。詳細計画の目的は、リファレンスのルールをファイルごとの具体的な設計に落とし込み、実行フェーズでの判断を不要にすることである。

### 手順

以下の順序でリファレンスを読み込み、ファイルごとの詳細計画を作成する。作成モードではすべての順序を実行する。修正モードでは変更対象のファイルに該当する順序のみ実行する。

| 順序 | 詳細計画ファイル | 読み込むリファレンス |
|---|---|---|
| 1 | `01-chart-yaml.md` | chart-files.md, general-conventions.md |
| 2 | `02-values-yaml.md` | values.md, pods-basic.md, pods-security.md, pods-reliability.md |
| 3 | `03-helpers.md` | templates.md, labels.md |
| 4 | `04-workload.md` | templates.md, pods-basic.md, pods-security.md, pods-reliability.md |
| 5 | `05-service.md` | templates.md |
| 6 | `06-ingress.md`（該当時） | templates.md |
| 7 | `07-rbac.md`（該当時） | templates.md, rbac-basic.md, rbac-least-privilege.md |
| 8 | `08-other-resources.md`（該当時） | templates.md |
| 9 | `09-documentation.md` | documentation.md |
| 10 | `10-environments.md` | values-environment.md |

詳細計画は `results/helm-chart-plan-{yyyyMMddHHmm}/` ディレクトリに保存する。

### 各詳細計画に記載する内容

リファレンスのルールを抽象的に参照するのではなく、そのファイルに適用した結果を書く。

- **作成するファイルパス**
- **リファレンスから導出した具体的な設計**: キー名と値、テンプレート構造、ラベル構成、securityContext のフィールドと値、probes の種類と設定値など
- **ユーザー指定値とリファレンスのデフォルト値の対応**: ヒアリングで得た値がどこに反映されるか

各ステップではリファレンスを読み込んでから詳細計画を書き、完了してから次のステップに進む。

## 検証手順

作成モード・修正モード共通で、実行後に以下を順に実行する。

1. **`helm lint <chart-dir>`** — chart 構造とフォーマットの静的チェック
   - WARNING も含めてすべて確認する
   - `icon` 未設定の WARNING は無視してよい
2. **`helm template <release-name> <chart-dir>`** — テンプレートのレンダリング確認
   - レンダリングエラーがないことを確認する
   - 出力されたマニフェストの内容が詳細計画と一致することを確認する
3. **`helm template <release-name> <chart-dir> -f <chart-dir>/values-{env}.yaml`** — 環境別 values でのレンダリング確認
   - 環境別 values ファイルが存在する場合、それぞれで実行する
   - 環境別の差分が意図通りに反映されていることを確認する

エラーが発生した場合はファイルを修正し、再度検証を実行する。すべてのチェックをパスするまで繰り返す。

## リファレンス読み込みルール

`references/` 配下のファイルは作業内容に応じて必要なものだけを読み込む。一括読み込みはしない。

### 作成モード・修正モード

「詳細計画の作成手順」セクションのテーブルに従い、ファイルごとに必要なリファレンスを読み込む。`general-conventions.md` はすべての作業で最初に読み込む。

### レビューモード

「レビュー実行」セクションの処理順序に従い、リファレンスを 1 つずつ読み込む。全リファレンスに対して準拠/違反/対象外の判定を行うため、省略しない。
