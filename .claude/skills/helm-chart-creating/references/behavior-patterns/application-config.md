# 振る舞いパターン: アプリ設定ファイル生成

アプリが独自の設定ファイル (rsyslog.conf, nginx.conf, fluentd.conf, application.yaml 等) を必要とする場合の Phase 1 ヒアリングと生成プロセス。

## 該当判定

以下のいずれかに該当する場合:
- アプリが独自設定ファイルを読む必要がある (rsyslog, fluentd, nginx, haproxy, envoy, jvm, 各種ミドルウェア等)
- ユーザーが「設定ファイルを ConfigMap で渡す」 等を言及
- アプリ特性から設定ファイルが必須と判断される

## ワークロード別の適用

| WL | 該当度 |
|---|---|
| Deployment | ✅ よくある |
| DaemonSet | ✅ よくある (rsyslog, fluentd, vector 等) |
| StatefulSet | ✅ よくある (DB 設定等) |
| Job / CronJob | △ ある (バッチアプリの設定) |

---

## 大原則 (Phase 1 で確定する)

**設定ファイルは Phase 1 のヒアリング対象**である。Phase 2 計画書で初出してはならない (後出し禁止)。

Phase 1 の終了時点で、設定ファイルの **中身まで** ユーザーと擦り合わせ済みでなければ Phase 1 は完了しない。

---

## 提供方式の確認 (Phase 1 必須)

設定ファイルが必要と判明した時点で、以下を能動的に確認する:

```
このアプリは <設定ファイル名> が必要です。以下のいずれの方式で進めますか?

A) 既存の設定ファイル を提供する
   → そのファイルの内容を貼り付けてください

B) 意図 (要件) を伝えて、私が設定ファイルを生成する
   → 受信プロトコル/ポート、フィルタ条件、転送先、エラー処理、ロギング方針等を教えてください

C) お任せ (最小限のサンプルから始めて、運用後に編集)
   → アプリのデフォルト設定に近い最小構成で生成します
```

---

## 方式 A: 既存ファイル提供

ユーザーから提供された設定ファイルをそのまま使う。Claude は **変更しない** (Phase 1 でフォーマットや明らかな構文エラーの確認はしてよいが、意図的な内容変更はしない)。

ConfigMap に格納する形でテンプレートに組み込む (`config-mount.md` 参照)。

---

## 方式 B: 意図ベース生成 (重要)

ユーザーが意図 (logic/spec) を提供する場合、Claude が対象ソフトの構文に翻訳する。これは deployment.yaml 生成と同質の作業で、本スキルの範疇内。

### 必須プロセス

1. **意図のヒアリング** (Phase 1 で完結):
   - 必要な情報を全て聞き出す (受信プロトコル、ポート、フィルタ条件、転送先、エラー処理、ロギング方針 等)
   - 意図が曖昧な部分は追加質問する

2. **公式構文ドキュメントを `web_fetch` で確認** (記憶ベース禁止):
   - rsyslog: https://www.rsyslog.com/doc/
   - fluentd: https://docs.fluentd.org/
   - nginx: https://nginx.org/en/docs/
   - その他の対象ソフトの公式ドキュメント
   - **記憶で書かない**。バージョンによって構文が変わるため

3. **前提を明示**:
   - 対象ソフトのバージョン (例: 「rsyslog v8 系、RainerScript 構文で生成」)
   - 使用する構文スタイル (rsyslog なら RainerScript / property-based / legacy のどれか)
   - 使用するモジュール (rsyslog なら `imudp`, `omfwd` 等)

4. **代替案がある場合は提示**:
   - 例 (rsyslog): 「シンプルな if-then で書く / mmnormalize モジュールで構造化パースする」
   - 例 (nginx): 「server_name で振り分ける / location で振り分ける」

5. **生成案をユーザーに提示し、Phase 1 内で擦り合わせる**:
   - 生成内容を表示
   - 「この内容で OK ですか? 修正点があれば教えてください」 と確認
   - **承認されるまで Phase 1 は終わらない**

6. **Phase 1 終了時点で確定**: 計画書では確定済みの設定ファイル中身を「7. 設定ファイル」 セクションにそのまま載せる (新規承認プロセス不要)

---

## 方式 C: お任せ

Claude が対象ソフトのデフォルト相当の最小構成を生成する。ただし:

1. **公式の minimum example を `web_fetch`** で確認
2. **「最小構成です。実際の運用に応じて編集してください」 と明示**
3. 生成内容を Phase 1 でユーザーに見せ、確認を取る

---

## 計画書 (Phase 2) での扱い

Phase 1 で確定した設定ファイル中身を「7. 設定ファイル」 セクションに載せる。その際:

- **前提** (バージョン、構文スタイル、モジュール等) を併記
- **新規承認プロセスは不要** (計画書全体承認の一部)

```markdown
### 7. 設定ファイル

**ファイル**: `rsyslog.conf`
**前提** (Phase 1 で擦り合わせ済み):
- rsyslog v8 系、RainerScript 構文
- 使用モジュール: `imudp`, `omfwd`
- フィルタ: `if re_match($rawmsg, '^<[0-9]+>1 ') then` で RFC5424 判定

**内容**:
\`\`\`
(rsyslog.conf の中身)
\`\`\`
```

---

## よくある落とし穴 (Phase 2 セルフチェック対象)

- **計画書で初めて設定ファイル中身を提示**: 後出し禁止
- **Claude が記憶ベースで構文を書く**: バージョン差で動かない可能性。公式 `web_fetch` 必須
- **対象ソフトのバージョンを前提に明示しない**: 後でユーザーが別バージョンで使うと動かない
- **代替案を提示しない**: ユーザーが知らないだけで、別の書き方が適切な可能性
- **「お任せ」 で過度に複雑な設定を生成**: 最小から始めるべき
- **設定ファイルの ConfigMap マウント方法を考慮しない**: `config-mount.md` の `subPath` 使用時の挙動に注意 (ConfigMap 更新が反映されない)
