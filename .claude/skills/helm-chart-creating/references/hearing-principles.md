# ヒアリング原則

Phase 1 のヒアリングは機械的なテンプレートではなく、本ファイルの**原則**に沿って Claude がアプリ特性から能動的に質問を組み立てる。

## 想定ユーザー

アプリエンジニア。k8s 基本概念は理解、Helm chart ベストプラクティスには非精通。

## 目的

ユーザーと **並走** しながら helm chart 構成を決める。Claude は受動的な質問機ではなく、ベストプラクティスを知る伴走者として能動的に提案する。

---

## 原則 1: 能動的提案 (最重要)

**Claude は受動的に「未設定です」 と残してはならない**。アプリ特性から判断して必要な設定を**先に提案** し、ユーザーに「これでよいか」 を確認する形でヒアリングする。

### 受動的ヒアリングと能動的ヒアリングの違い

**❌ 受動的 (禁止)**:
> 「nodeSelector はどうしますか?」
> 「priorityClassName を設定しますか?」
> 「liveness probe は必要ですか?」

**✅ 能動的 (推奨)**:
> 「rsyslog DaemonSet で全ノードからログ収集するとのことなので、以下を提案します:
> - **PriorityClassName**: ログ収集はノード上で確実に動いてほしいインフラなので、`system-node-critical` または専用の高優先度クラスを推奨します
> - **tolerations**: master ノードにも配置しますか? それともワーカーノードのみで OK ですか?
> - **liveness probe**: rsyslog は UDP ベースなので HTTP probe が使えません。`rsyslogd.pid` を使った exec probe を提案します
>
> この方針で進めて良いですか?」

### 能動提案の手順

1. ユーザーリクエストから **アプリの種類** を抽出する (ログ収集、Web API、バッチ処理 等)
2. ワークロード種別を確定する (推論禁止、必ず質問)
3. アプリ特性から該当する **behavior-pattern** を抽出する
4. 各 behavior-pattern の **「能動的提案」** セクションを実行する
5. **複数 behavior-pattern を 1 メッセージにまとめて** 提示する (ユーザーの認知負荷を下げる)
6. ユーザー応答を待つ
7. 応答から新たな behavior-pattern が判明したら追加で読み込んで、再び能動提案

---

## 原則 2: ワークロード未明示なら必ず質問する (推論禁止)

```
ワークロード種別を教えてください:
① Deployment (ステートレス Web/API)
② DaemonSet (全ノード常駐エージェント)
③ StatefulSet (永続化や安定ネットワーク ID 必要)
④ Job (一回限りバッチ)
⑤ CronJob (定期スケジュールバッチ)
```

**理由**: ワークロード選定は設計根幹。間違えると作り直し。キーワード推論は脆い (例: 「rsyslog で受信」 は Deployment にも DaemonSet にもなり得る)。質問コストは低い。

**唯一の例外**: ユーザーが英単語として明示記述している場合のみ採用。

---

## 原則 3: k8s 技術用語を直接聞かない

k8s リソース (ConfigMap, Secret, PVC, Service 等) の選択は **Claude が判断** する。ユーザーには **アプリ機能のレベル** で質問する。

### NG 質問例

- ❌「ConfigMap と Secret のどちらでマウントしますか?」
- ❌「PVC のアクセスモードは ReadWriteOnce と ReadWriteMany のどちらですか?」
- ❌「Service の type は ClusterIP と LoadBalancer のどちらですか?」
- ❌「nodeSelector と nodeAffinity のどちらを使いますか?」

### OK 質問例

- ✅「設定値に機密情報 (パスワード等) は含まれますか?」 → Claude が ConfigMap/Secret を判断
- ✅「データを複数の Pod から同時に読み書きする必要はありますか?」 → Claude がアクセスモードを判断
- ✅「外部 (インターネット/社内ネットワーク) からアクセスしますか?」 → Claude が公開方式を判断
- ✅「特定のノード (GPU 搭載、Spot 等) で動かす必要はありますか?」 → Claude が配置を判断

---

## 原則 4: 設定ファイル中身は Phase 1 で確定する

アプリ設定ファイル (rsyslog.conf, nginx.conf 等) が必要な場合、その**中身まで** Phase 1 で確定する (`behavior-patterns/application-config.md` 参照)。

**禁止事項**: 計画書 (Phase 2) で初めて設定ファイル中身を提示すること。これは「後出し」 でワークフロー違反。

---

## 原則 5: ユーザー入力は仮置き禁止

ユーザーが提供すべき情報 (image.repository、image.digest 等) を「後ほど記述」 等の仮置きで進めない。

- ユーザーが今すぐ提供できない場合は、明示的に **保留** 扱いとし、計画書の冒頭に **「保留項目リスト」** として記載する
- Phase 3 の chart 生成は保留項目があっても実施可能 (空欄を残す形で)
- ただし計画書承認時に「保留項目があります、後で必ず埋めてください」 と明示する

---

## 中継・転送系アプリの扱い

rsyslog、fluentd、vector、logstash 等の中継・転送系アプリは、用途パターンによって最適なワークロードが変わる。ワークロード質問と同時に **用途パターン** も確認する:

```
このアプリの用途を教えてください:
A. 各ノードのログを収集して集約サーバへ送る → DaemonSet が一般的
B. 集約サーバとして各ノードからのログを受け取る → Deployment が一般的
C. 中継サーバとして外部から受信したログを別の外部に転送する → Deployment または DaemonSet (受信特性次第)
D. その他 (具体的に教えてください)
```
