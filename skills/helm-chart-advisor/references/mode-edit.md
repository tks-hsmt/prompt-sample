# 編集モード

既存の Helm chart を修正する場合の手順。

## ヒアリング

作業を始める前に、以下の情報がユーザーの依頼から読み取れるか確認する。

**必須（不明なら必ず確認）:**
- **対象ファイル**: どのチャートのどのファイルを変更するか（パスが明確でなければ確認）
- **変更内容**: 何をどう変えたいか（追加・修正・削除）

**曖昧になりやすいポイント（依頼内容から判断できなければ確認）:**
- **変更スコープ**: 依頼された箇所だけを修正するか、関連するベストプラクティス違反（命名規則、セキュリティ設定等）も合わせて修正するか
- **既存パターンとの整合性**: 既存チャートが独自の構造やパターンを持っている場合、それに合わせるかベストプラクティスに寄せるか

## 変更内容の確認

ヒアリングで情報が揃い、既存チャートの構成を確認したら、作業着手前に変更内容をユーザーに提示する：

1. **変更する箇所の一覧と挙動の説明**: どのファイルのどの部分をどう変更し、変更後にどのような挙動になるかを説明する。変更しない箇所も含め、変更後の全体像がわかるようにする

2. **不整合チェック**: 変更を適用した結果、既存の構成と矛盾が生じないかを確認する。例:
   - Ingress を追加するが Service の port と Ingress の backend port が不一致
   - プローブのポートがコンテナのポートと異なる
   - resources を追加するが HPA の targetCPU と requests のバランスが不適切

3. **ベストプラクティス違反への対応**（変更スコープの確認時に「合わせて修正する」となった場合）: 追加で修正する項目とその理由

ユーザーの OK を得てから作業に着手する。変更が軽微で影響が自明な場合（コメント追加、タイポ修正等）はこのステップを省略してよい。

## 手順

1. まず既存のファイル構成を確認する（Chart.yaml, values.yaml, templates/ 配下）
2. 変更箇所に関連するリファレンスを参照し、ベストプラクティスを適用する
3. 既存の values 構造やテンプレートパターンとの一貫性を維持する
4. 変更後に `helm lint` と `helm template` で検証する

## 変更箇所とリファレンスの対応

| 変更対象 | 参照ファイル |
|---|---|
| Chart.yaml（バージョン、メタデータ） | `chart-structure.md` |
| values.yaml（パラメータ追加・変更） | `values.md` |
| テンプレート（構造、フォーマット） | `templates.md` |
| 依存チャート、CRD | `dependencies.md` |
| securityContext、NetworkPolicy、ServiceAccount | `security.md` |
| イメージ、リソース、プローブ、ラベル | `workloads.md` |
| デプロイ運用、CI/CD、テスト、シークレット | `operations.md` |
| OCI、Helmfile、ArgoCD、ドキュメント | `ecosystem.md` |

変更箇所に関連するファイルだけを読むこと。全ファイルを一括で読み込まない。
